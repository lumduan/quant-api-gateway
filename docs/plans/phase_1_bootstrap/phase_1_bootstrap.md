## Objective

Stand up the FastAPI project skeleton with Docker Compose so that
`docker compose up -d` works on a fresh clone, the container joins
`quant-network`, and `GET /health` returns `200 OK`.

This is the foundation every later phase (data models, ingestion, aggregation,
caching, REST API, ops/quality gate) depends on. No Phase 1 deliverable
introduces business logic — only structure and a healthcheck.

## Scope

### In scope

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| `quant-network` missing locally → compose fails | README documents `docker network create quant-network`; `external: true` makes the failure loud |
| `Settings()` at import time crashes tests | Expose `get_settings()` (cached with `functools.lru_cache`); tests set env via `monkeypatch.setenv` |
| `mypy --strict` flags zero-arg `Settings()` call | Confine the call to `get_settings()`; one `# type: ignore[call-arg]` if needed |
| `python:3.12-slim` lacks `curl` for healthcheck | `apt-get install -y --no-install-recommends curl` in the runtime stage |
| Replacing `src/main.py` orphans `tests/test_main.py` | Replace both in the same commit |
| Empty `__init__.py` files drop coverage | `coverage.py` treats zero-statement files as 100%; confirmed by `--cov-report=term-missing` |
| Renaming project breaks hatchling wheel build | `tool.hatch.build.targets.wheel.packages = ["src"]` references the directory, not the project name |

### Decisions taken vs. the plan

- **Lifespan testing**: `httpx.ASGITransport` does **not** dispatch ASGI
  lifespan events (it only routes HTTP scope). The
  `test_lifespan_runs_startup_and_shutdown` test instead enters the
  `@asynccontextmanager`-decorated `lifespan` directly with `async with
  lifespan(app): pass`, which is the cleanest way to cover both branches
  without adding `asgi-lifespan` as a dev dependency. If a later phase
  needs lifespan-dispatched fixtures (e.g. for the asyncpg pool), revisit
  `asgi-lifespan.LifespanManager`.
- **Lifespan-context identity check dropped**: FastAPI wraps the
  user-supplied lifespan in `_merge_lifespan_context`, so
  `app.router.lifespan_context is lifespan` is never true. The test was
  replaced by the direct-context test above.
- **`curl` in the runtime stage**: kept the ROADMAP healthcheck command
  (`curl -f http://localhost:8000/health || exit 1`) verbatim by installing
  `curl` in the runtime stage of the Dockerfile (`apt-get install -y
  --no-install-recommends curl` + `rm -rf /var/lib/apt/lists/*`). The
  alternative was a Python healthcheck (`python -c "import httpx; ..."`)
  which would have spun up a Python interpreter every 30 s; `curl` is
  cheaper at runtime.
- **`requires-python = ">=3.11"` retained**: the existing CI matrix tests
  both 3.11 and 3.12 (`.github/workflows/ci.yml`). The ROADMAP showed
  `python:3.12-slim` for the Docker base image but did not require dropping
  3.11 support, and there's no reason to lock out 3.11 developers yet.
- **`Field(min_length=1)` on `internal_api_key`**: ROADMAP doesn't specify,
  but an empty API key should not be accepted — added a min-length
  constraint and a regression test (`test_settings_rejects_empty_api_key`).
- **`get_settings()` cached with `functools.lru_cache`**: avoids the
  hidden-global anti-pattern listed in
  `.claude/memory/anti-patterns.md`. Tests call
  `get_settings.cache_clear()` in the autouse `set_env` fixture to keep
  per-test state isolated.

### Deviations from the plan

- **Docker stack verification deferred**: the Docker daemon was not
  running on the implementation host (`docker info` failed). `docker
  compose config` validates the YAML and `docker compose version` reports
  CLI v5.1.3, but the live `docker compose up -d` end-to-end run must be
  done by a user with Docker available. Three acceptance-criteria boxes
  remain unchecked above and call this out explicitly. The Dockerfile and
  `docker-compose.yml` are byte-for-byte aligned with the ROADMAP §1.3
  template aside from the explicit `curl` install in the runtime image.

### Anti-patterns avoided / patterns established

- `get_settings()` lazy + cached, rather than a module-level
  `settings = Settings()` — keeps tests deterministic, avoids
  import-time-crash on missing env vars.
- ASGITransport-based async tests instead of `TestClient` — pure async,
  no threadpool, no lifespan side effects in the test transport layer.

Nothing new for `.claude/memory/anti-patterns.md` — every issue
encountered was already documented there.
